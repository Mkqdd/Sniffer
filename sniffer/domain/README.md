# 基于DNS-流关联的智能网络流量分类系统

## 项目概述

本项目实现了一个创新的网络流量分析与分类系统，通过DNS-流关联技术和AC自动机域名匹配算法，实现了对PCAP文件中网络流量的智能拆分、关联和分类。该系统能够自动识别流量所属的应用类别，为网络流量分析、安全检测和应用识别提供了高效的技术方案。

**新增功能**：现已支持将分析结果直接写入 FastMonitor 数据库，实现仪表盘实时显示和增量更新。

## 核心创新点

### 创新点一：DNS-流关联技术（DNS-Flow Correlation）

#### 1.1 技术原理

传统的网络流量分析通常基于五元组（源IP、目标IP、源端口、目标端口、协议）进行流识别，但这种方法无法直接获取应用层语义信息。本系统创新性地提出了DNS-流关联技术，通过分析DNS查询和响应，建立域名到IP地址的映射关系，然后反向关联网络流到对应的域名。

**技术流程：**
1. **DNS信息提取**：从PCAP文件中提取所有DNS查询和响应包
2. **域名-IP映射构建**：解析DNS响应中的A记录，建立 `domain → {IPs}` 的映射
3. **IP-域名反向映射**：构建 `IP → [domains]` 的反向查找表
4. **流-域名关联**：通过流的源IP或目标IP在反向映射表中查找对应的域名

#### 1.2 技术创新性

- **双向关联机制**：不仅支持通过目标IP查找域名（服务器端），还支持通过源IP查找域名（客户端），提高了关联的准确性和覆盖率
- **多域名支持**：一个IP可能对应多个域名（CDN、负载均衡等场景），系统能够识别并关联所有相关域名
- **DNS包关联**：将相关的DNS查询/响应包与数据流合并，保持流上下文的完整性

### 创新点二：基于后缀AC自动机的域名匹配算法

#### 2.1 算法设计

传统的域名匹配通常使用前缀匹配或简单的字符串匹配，无法处理域名后缀匹配的场景（如 `*.example.com` 匹配 `sub.example.com`）。本系统实现了基于后缀AC自动机的高效域名匹配算法。

**核心思想：**
- **域名反转**：将域名从TLD（顶级域名）开始构建Trie树，例如 `example.com` 转换为 `["com", "example"]`
- **AC自动机优化**：在Trie树基础上构建失败指针，实现O(n)时间复杂度的多模式匹配
- **最长匹配原则**：选择匹配长度最长的域名作为分类结果，确保分类的准确性

#### 2.2 算法优势

1. **高效匹配**：AC自动机的时间复杂度为O(n+m)，其中n是文本长度，m是所有模式的总长度
2. **多模式匹配**：一次遍历可以匹配所有可能的域名模式
3. **后缀匹配**：支持通配符域名匹配，如 `*.google.com` 可以匹配 `www.google.com`、`mail.google.com` 等
4. **优先级处理**：通过域名长度和文件名优先级，选择最具体的分类结果

### 创新点三：智能流合并策略

#### 3.1 合并算法

系统实现了两层次的流合并策略：

**第一层：按域名分组**
- 使用第一个域名作为分组key（转换为小写）
- 合并所有具有相同第一个域名的流的数据包
- 去重处理，避免重复数据包

**第二层：按IP对分组**
- 对于没有域名的流，按 `src_ip_dst_ip` 分组
- 合并相同IP对的所有流

#### 3.2 技术优势

- **减少文件数量**：避免相同域名的多个流产生重复文件
- **保持时间顺序**：合并后的数据包按时间戳排序，保持流的时序性
- **内存优化**：使用指针集合进行去重，避免数据包复制

### 创新点四：双向流统一处理机制

#### 4.1 统一化算法

系统通过FlowKey的统一排序机制，确保双向流（A→B 和 B→A）使用相同的标识符：

```cpp
if (s_ip < d_ip || (s_ip == d_ip && s_port < d_port)) {
    // 保持原顺序
} else {
    // 交换源和目标
}
```

#### 4.2 技术意义

- **避免重复处理**：确保每个双向流只处理一次
- **一致性保证**：双向流的数据包合并到同一个文件中
- **性能优化**：减少50%的流处理开销

### 核心数据结构

#### FlowKey（流标识）
```cpp
struct FlowKey {
    std::string src_ip;      // 源IP（统一排序后）
    std::string dst_ip;      // 目标IP（统一排序后）
    uint16_t src_port;       // 源端口（统一排序后）
    uint16_t dst_port;       // 目标端口（统一排序后）
    uint8_t protocol;        // 协议类型（TCP/UDP）
};
```

**特点：**
- 双向流统一：通过排序确保A→B和B→A使用相同的key
- 支持map排序：实现了 `<` 操作符用于std::map排序

#### DomainGroup（域名组）
```cpp
struct DomainGroup {
    std::set<std::string> domains;           // 所有域名（去重）
    std::vector<PacketInfo> all_packets;     // 所有数据包
    std::set<pcpp::RawPacket*> packet_set;   // 用于去重
    std::string best_category;               // 最佳分类结果
    int flow_count;                          // 包含的流数量
};
```

**特点：**
- 域名集合：存储所有相关域名
- 数据包合并：合并多个流的数据包
- 去重机制：使用指针集合避免重复

#### ACNode（AC自动机节点）
```cpp
struct ACNode {
    std::map<std::string, ACNode*> children;  // 子节点（域名部分→节点）
    ACNode* fail;                             // 失败指针
    std::vector<std::pair<std::string, int>> outputs;  // 输出（文件名，域名长度）
};
```

**特点：**
- 后缀Trie：从TLD开始构建
- 失败指针：实现高效的字符串匹配
- 多输出支持：一个节点可以匹配多个模式

## 关键技术细节

### 1. DNS信息提取算法

**算法流程：**
1. 遍历PCAP文件中的所有数据包
2. 识别DNS层数据包（`packet.isPacketOfType(pcpp::DNS)`）
3. 提取DNS查询中的域名（`dns_layer->getFirstQuery()->getName()`）
4. 解析DNS响应中的A记录（`dynamic_cast<pcpp::IPv4DnsResourceData*>`）
5. 建立 `domain → {IPs}` 映射关系

**关键代码：**
```cpp
// 只提取A记录（IPv4），忽略AAAA记录（IPv6）
if (answer->getDnsType() == pcpp::DNS_TYPE_A) {
    auto* ipv4_data = dynamic_cast<pcpp::IPv4DnsResourceData*>(dataPtr.get());
    if (ipv4_data != nullptr) {
        std::string ip = ipv4_data->getIpAddress().toString();
        dns_info.domain_to_ips[domain].insert(ip);
    }
}
```

### 2. 流-域名关联算法

**算法流程：**
1. 构建IP到域名的反向映射：`IP → [domains]`
2. 对于每个流，检查源IP和目标IP
3. 在反向映射表中查找对应的域名列表
4. 合并所有找到的域名（去重）

**时间复杂度：** O(1) 查找（使用std::map）

### 3. AC自动机构建算法

**构建步骤：**
1. **插入阶段**：将域名反转后插入Trie树
   - `example.com` → `["com", "example"]`
   - 在终止节点记录分类标签和域名长度

2. **失败指针构建**（BFS算法）：
   ```cpp
   ACNode* fail = current->fail;
   while (fail != root && fail->children.find(key) == fail->children.end()) {
       fail = fail->fail;  // 沿着失败指针回溯
   }
   ```

3. **输出合并**：将失败指针链上的所有输出合并到当前节点

**匹配算法：**
- 时间复杂度：O(n)，其中n是域名长度
- 空间复杂度：O(m)，其中m是所有模式的总长度

### 4. 最长匹配策略

**匹配原则：**
- 遍历域名的所有可能后缀
- 选择匹配长度最长的结果
- 如果多个结果长度相同，选择文件名更具体的（不包含`category-`前缀）

**示例：**
- 域名：`www.example.com`
- 模式1：`example.com`（长度2）
- 模式2：`www.example.com`（长度3）
- 结果：选择模式2（更长）

### 5. 流合并与去重算法

**合并策略：**
1. **按域名分组**：使用第一个域名作为分组key
2. **数据包合并**：合并所有流的数据包到同一个组
3. **去重处理**：使用`std::set<pcpp::RawPacket*>`进行指针去重
4. **时间排序**：按时间戳排序，保持流的时序性

**去重机制：**
```cpp
std::set<pcpp::RawPacket*> packet_set;
if (packet_set.find(pkt_info.raw_packet) == packet_set.end()) {
    all_packets.push_back(pkt_info);
    packet_set.insert(pkt_info.raw_packet);
}
```

## 性能优化技术

### 1. 内存管理优化
- **RawPacket副本**：使用指针管理数据包副本，避免大量数据复制
- **智能指针去重**：使用指针集合进行去重，O(1)查找复杂度
- **及时释放**：处理完成后统一释放内存

### 2. 算法复杂度优化
- **AC自动机**：O(n)时间复杂度的多模式匹配
- **Map查找**：O(log n)的IP-域名查找
- **流合并**：O(n)的线性合并，n为流数量

### 3. 文件I/O优化
- **单次遍历**：DNS信息提取和数据包分组在一次遍历中完成
- **批量写入**：合并后的数据包批量写入文件
- **目录预创建**：提前创建分类目录，避免重复检查

## 使用方法

### 编译
```bash
g++ -std=c++17 -Wall -g \
    -I/path/to/pcapplusplus/include \
    split.cpp -o split \
    -L/path/to/pcapplusplus/lib \
    -lPacket++ -lPcap++ -lCommon++ -lpcap -lpthread
```

### 运行
```bash
./split --input test.pcap --output output \
        --data-dir domain-list-community/data \
        --entity-map domain-list-community/entity_map.json
```

### 输出格式
```
文件名,标签
例如：
lf-rc1.yhgfb-cn-static.com.pcap,bytedance
addons.mozilla.org.pcap,mozilla
192.168.152.131_192.168.152.2.pcap,unknown
```

### 输出目录结构
```

├── classified/          # 已分类的流
│   ├── bytedance/      # 字节跳动相关流量
│   ├── mozilla/        # Mozilla相关流量
│   ├── google/         # Google相关流量
│   └── ...
└── unknown/            # 未分类的流
    ├── domshow.vhostgo.com.pcap

```
